# Шпаргалка — команды для ручного выполнения

> Команды, которые Cline не выполняет сам из-за известных багов терминала. 
> Выполнять вручную в терминале VS Code.

## Git

**Проверить статус:**
git status

**Закоммитить изменения:**
git add .
git commit -m "короткое описание изменений"

**Просмотреть историю коммитов:**
git log --oneline

**Работа с ветками:**
git checkout -b feature/название-ветки     # создать и перейти
git checkout master                         # вернуться на master
git merge feature/название-ветки            # слить ветку в master
git branch -d feature/название-ветки        # удалить отработанную ветку

**Запушить на GitHub:**
git push

**Отменить незакоммиченные изменения:**
git restore .

## VPS — просмотр состояния сервера (без Claude Code)

> Для случая, если пропал доступ к Claude Code, кончились лимиты, или просто
> хочется своими глазами посмотреть, что происходит на сервере. Все команды
> ниже — **только чтение**, ничего не меняют и не могут сломать бота.
> Выполнять в PowerShell на своём компьютере (не нужен Linux/WSL — обычная
> команда `ssh` в Windows уже встроена).

### 1. Подключиться к серверу

```
ssh root@46.62.155.17
```

Ключ уже лежит на компьютере (`C:\Users\DM_PC\.ssh\id_ed25519`), спрашивать
пароль не должно — просто открывается новая строка вида `root@arbitrage-bot-prod:~#`.
Это значит, что подключение открыто и дальше можно вводить команды с сервера.
Чтобы выйти — набрать `exit` и нажать Enter.

Если вместо этого ошибка `Connection timed out` — сервер недоступен по сети
(это уже не решается локально, писать нам). Если `Permission denied` — что-то
не так с ключом, тоже писать нам, самостоятельно не чинить.

### 2. Статус бота

```
systemctl status multi-exchange-arbitrage.service
```

**Нормально:** зелёная точка и строка `Active: active (running)`.
**Плохо:** `Active: failed` или `inactive (dead)` — бот не работает, зафиксировать
(скопировать весь вывод команды в заметку) и написать нам. Самостоятельно не
перезапускать (см. раздел "Чего не делать" ниже).

Чтобы выйти из просмотра статуса (если экран "залип") — нажать `q`.

### 3. Живые логи в реальном времени

```
journalctl -u multi-exchange-arbitrage.service -f
```

Строки будут появляться сами по мере работы бота — это нормально, значит бот
активен. Больше всего интересны строки со словом `ERROR` (проблема) или
`СУТОЧНАЯ СВОДКА` (ежедневный отчёт с числами: сколько возможностей найдено,
сколько сделок открыто/закрыто). Чтобы остановить просмотр (сам бот при этом
НЕ останавливается) — нажать `Ctrl+C`.

### 4. lnav — удобный просмотр логов

```
lnav /root/multi-exchange-arbitrage/logs/systemd-stdout.log
```

Откроется текстовый интерфейс с логами. Полезные клавиши внутри:
- `/` — поиск по тексту (например, набрать `/ERROR` и Enter, чтобы найти ошибки)
- `n` / `N` — перейти к следующему/предыдущему найденному совпадению
- `q` — выйти обратно в обычный терминал

### 5. Сколько места на диске

```
df -h
```

Смотреть на строку `/dev/sda1` (основной диск) — колонка `Use%` показывает,
сколько занято в процентах. **Нормально** — до ~70-80%. Если ближе к 100% —
писать нам, диск может забиться раньше срока.

### 6. Работает ли база данных

```
docker ps
```

**Нормально:** в списке есть строка `arbitrage-db` со статусом `Up ...`
(например `Up 3 days`). Если её нет или написано `Exited` — база не работает,
писать нам.

Если хочется заглянуть в логи самой базы (реже нужно):
```
docker compose -f /root/multi-exchange-arbitrage/docker-compose.yml logs timescaledb --tail 50
```

### 7. Быстрые вопросы к данным напрямую (без дашборда)

Есть отдельный пользователь базы данных `arbitrage_readonly` — он **умеет
только читать**, ничего испортить не может, даже если ошибиться в команде.
Пароль этой роли **не хранится в этом файле** (файл публикуется в открытый
git-репозиторий) — храни его отдельно (например, в своём менеджере паролей),
он был показан один раз при создании роли. Ниже — команды с плейсхолдером
`<ПАРОЛЬ>`, подставлять свой при копировании.

**Сколько арбитражных возможностей найдено за последний час:**
```
docker exec -e PGPASSWORD=<ПАРОЛЬ> arbitrage-db psql -U arbitrage_readonly -d arbitrage -c "SELECT COUNT(*) FROM arbitrage_opportunities WHERE timestamp > extract(epoch from now() - interval '1 hour');"
```

**Сколько симулированных сделок открыто прямо сейчас:**
```
docker exec -e PGPASSWORD=<ПАРОЛЬ> arbitrage-db psql -U arbitrage_readonly -d arbitrage -c "SELECT COUNT(*) FROM simulated_trades WHERE status='open';"
```

**Последние 10 закрытых сделок (что, где, какой результат):**
```
docker exec -e PGPASSWORD=<ПАРОЛЬ> arbitrage-db psql -U arbitrage_readonly -d arbitrage -c "SELECT close_readable_time, outcome, realized_profit_percent FROM simulated_trades WHERE status='closed' ORDER BY closed_at DESC LIMIT 10;"
```

Ответ придёт в виде простой таблицы прямо в терминале — цифры и есть ответ,
дополнительно ничего нажимать не нужно.

### 8. Визуальный дашборд (Metabase)

Открыть в браузере (с телефона или компьютера, из любой сети):
```
http://46.62.155.17:3000
```

При первом входе Metabase попросит создать свой личный логин/пароль — это

заходе. Дальше — обычный веб-сайт с графиками и таблицами, командная строка
не нужна.

### Чего НЕ делать самостоятельно

Если что-то выглядит тревожно — **зафиксировать** (скопировать текст ошибки
или вывод команды к себе в заметку) и написать нам, когда будет возможность.
Не пытаться исправить самостоятельно:

- **Не перезапускать бота** (`systemctl restart ...`) — если он упал, важно
  сначала понять причину, а не просто "включить заново".
- **Не трогать файрвол** (`ufw ...`) — можно случайно заблокировать себе же
  доступ к серверу.
- **Не выполнять `docker compose down`** (особенно с флагом `-v`) — это может
  остановить или удалить базу данных со всей историей.
- **Не менять файлы на сервере** (никаких `nano`, `vim`, `rm` и т.п.) — даже
  если кажется, что нашёлся баг, правки кода делаются локально через Claude
  Code/Codex и приезжают на сервер через `git`, не вручную на проде.
