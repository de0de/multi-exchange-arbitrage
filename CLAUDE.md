# CLAUDE.md — инструкции для Claude Code в этом проекте

## Первым делом

Прочитай `PLAN.md` в корне проекта — это единственный источник правды об архитектуре,
схеме БД, roadmap и известных ограничениях. Не полагайся на предположения о структуре
проекта — сверяйся с PLAN.md и с реальным кодом.

## Проверка git-доступа (сделать один раз, в начале первой сессии)

Правило "никогда не выполнять git-команды" в `.clinerules` (для Cline) существует из-за
конкретного бага терминальной интеграции именно Cline (PSReadLine/shell integration
зависает на любой git-команде в PowerShell). Это ограничение **не переносится сюда
автоматически** — у тебя может быть другой механизм работы с терминалом.

Проверь безопасно: выполни `git status`. Если команда отработала нормально (не зависла,
вернула вменяемый вывод) — можешь предлагать пользователю git-команды напрямую (с обычными
для Claude Code гарантиями: commit/push только с явного подтверждения пользователя, как
в остальных проектах). Если `git status` зависла или повела себя странно — сообщи об этом
пользователю и предложи придерживаться того же ручного workflow, что описан для Cline
(предлагать команду текстом, пользователь выполняет сам).

## Формат задач

- **Точечные правки** (конфиги, PLAN.md, мелкие фиксы) — пользователь даёт точную
  спецификацию, выполняй буквально.
- **Разработка нового функционала с нуля** (новая биржа, новый модуль) — пользователь
  даёт краткое ТЗ, предлагай реализацию по шаблону из раздела 4 PLAN.md ("Добавление
  новой биржи"), пользователь ревьюит результат.
- Перед финальным отчётом о выполненной работе — сверяй реальное состояние файлов
  (прочитай файл / поищи метод), а не полагайся на память о том, что делал в рамках
  задачи. Если в отчёте упоминается конкретный файл/метод как "созданный" или
  "изменённый" — подтверди это, прежде чем писать в отчёт.

## Редактирование PLAN.md

- Точечные правки конкретных фрагментов, не полная перезапись файла — если добавляешь
  большой новый раздел, указывай точно, после какого существующего текста его вставить.
- Если не уверен в текущем полном содержимом PLAN.md — сначала прочитай файл целиком,
  и только затем предлагай правку.
- Перед применением любой правки PLAN.md — покажи пользователю diff/превью изменений
  и дождись подтверждения, даже для мелких точечных правок.

## Коммиты

- Коммитить по завершении каждой логически законченной единицы работы (например,
  отдельно "добавлен API-клиент", отдельно "добавлен Collector", отдельно "интеграция
  в main.py"), а не всей сессии одним коммитом.
- При прерывании сессии на середине задачи — дописывать короткую пометку в PLAN.md,
  раздел 5.1, под соответствующим пунктом (например: "В процессе: API-клиент готов,
  Collector — нет, интеграция в main.py — нет").

## Архитектурные принципы проекта (см. также PLAN.md разделы 2–4)

- Единое соединение SQLite на процесс (`sqlite3.connect()`, WAL-режим) — не открывай
  отдельные соединения в новых модулях без необходимости.
- Новая биржа добавляется строго по шаблону раздела 4 PLAN.md: API-класс наследует
  `BaseExchangeAPI`, Collector наследует `BaseDataCollector`, регистрация в
  `main.py` через одинаковый паттерн для всех бирж.
- При создании репозиториев (`MarketRepository`, `OrderBookRepository` и т.д.) —
  используй **согласованный slug** биржи (`"binance"`, `"kucoin"`, без произвольных
  суффиксов вроде `"_spot"`) — рассинхрон slug'ов уже один раз создал дублирующиеся
  таблицы в этом проекте (см. PLAN.md, раздел 6, "OrderBookRepository slug").
- Перед `DROP TABLE` в этой БД — всегда проверяй содержимое (`SELECT *`) перед
  удалением, не полагайся на предположение, что таблица мусорная.
- Не читай и не пытайся анализировать содержимое всей БД целиком (`arbitrage_data.db`)
  напрямую как контекст для рассуждений — если нужен анализ данных, пиши и выполняй
  реальный SQL/Python код для агрегации, не пытайся "прочитать" таблицы построчно
  в контекст. Это касается как обычных задач, так и будущего Analysis Module
  (см. PLAN.md, раздел 5.3).

## Эстафета Claude Code ↔ Codex (многомодельная работа)

Над проектом попеременно работают Claude Code и Codex (правила Codex — в
`AGENTS.md`). Правила эстафеты:

- **Строго последовательно:** в один момент времени рабочее дерево правит одна
  модель. Передача состояния — через `HANDOFF.md` в корне проекта (файл в
  `.gitignore`, это рабочее состояние эстафеты, не документация).
- **`HANDOFF.md` пишется ДО начала многошаговой задачи**, не после остановки —
  чтобы файл переживал внезапный обрыв по лимитам. Содержимое: формулировка
  задачи, список шагов, ограничения ("не менять…", "коммит не делать");
  галочки обновляются по мере выполнения шагов.
- **Формулировки в HANDOFF.md — конкретные.** Не "шаг 2 начат", а "создан
  `gate_futures_api.py`, `fetch_trading_pairs()` не проверен, интеграция в
  `main.py` не начата" — следующая модель не должна додумывать состояние.
- **Закоммиченное = проверенная работа; незакоммиченное = текущий шаг в
  процессе.** Коммиты в эстафете делает Claude Code.
- **Чужую незакоммиченную работу перед коммитом обязательно:** прочитать
  diff целиком, прогнать проверку (компиляция + тестовый запуск), при
  необходимости поправить. В сообщение коммита добавлять соавторство
  написавшей код модели.
